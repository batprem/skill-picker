# `SelectionService._sync` — คำอธิบายแบบเข้าใจง่าย

> ไฟล์: `src/skill_picker/service.py:38`

`_sync` คือฟังก์ชันที่ทำหน้าที่ **"จัดของให้ตรงกัน"** ก่อนค้นหา skill ทุกครั้ง

## เปรียบเทียบให้เห็นภาพ

- skill ทุกอันต้องถูกแปลงเป็น **"ตัวเลขพิกัด" (vector)** เพื่อเอาไว้เทียบความเหมือนกับคำค้น
- การแปลงนี้ช้าและเปลือง เลยมีการ **เก็บผลไว้ในแคช (cache)** ไว้ใช้ซ้ำ
- `_sync` คือ **"คนเช็คสต็อก"** ที่คอยทำให้แคชตรงกับของจริงในคลัง (ฐานข้อมูล SQLite) เสมอ

## มันทำอะไรทีละขั้น

1. **ดูรายการ skill ทั้งหมดที่มีตอนนี้** — เอาแค่ชื่อกับคำอธิบายสั้น ๆ ไม่เอาเนื้อหาเต็ม
2. **เช็คทีละอันว่าต้องคำนวณใหม่ไหม** — ถ้าเจอกรณีใดกรณีหนึ่งนี้ ก็ต้องแปลงใหม่:
   - **ยังไม่เคยแปลง** (skill ใหม่)
   - **เนื้อหาถูกแก้** (ลายนิ้วมือ / hash ไม่ตรงของเดิม)
   - **เปลี่ยนตัวแปลง** (เปลี่ยน embedder)

   ถ้าไม่เข้าข้อไหนเลย → ใช้ของเก่าในแคชได้เลย ไม่ต้องเสียเวลาแปลงใหม่
3. **แปลงอันที่ต้องแปลง ทีเดียวเป็นชุด** — รวบยอดทำรอบเดียว (ไม่ทำทีละอัน) แล้วเก็บผลกลับเข้าแคช
4. **ทิ้งของที่ไม่มีแล้ว** — skill ไหนถูกลบออกจากคลัง ก็ลบพิกัดของมันออกจากแคชด้วย
5. **บันทึก + สร้างดัชนีค้นหาใหม่** — ถ้ามีการเปลี่ยนแปลง ก็เซฟลงฐานข้อมูล แล้วประกอบดัชนีสำหรับค้นหาขึ้นใหม่

## ใจความสำคัญ

- **ขี้เกียจอย่างฉลาด** — แปลงใหม่เฉพาะอันที่เปลี่ยน ส่วนที่เหมือนเดิมใช้ของเก่า เลยเร็ว
- **ตรงเสมอตอนค้นหา** — เพราะ `select()` เรียก `_sync` ก่อนทุกครั้ง การค้นหาจึงไม่มีทางเจอข้อมูลเก่าที่ไม่ตรงกับความจริง
- **เปลี่ยน embedder = ล้างแคชทั้งหมดอัตโนมัติ** — เพราะ "ตัวแปลง" เป็นส่วนหนึ่งของกุญแจแคช พอเปลี่ยนก็ถือว่าของเก่าใช้ไม่ได้ทันที

> พูดสั้น ๆ: **มันคือคนทำความสะอาดที่ทำให้แคชเป็นกระจกสะท้อนคลังของจริงเสมอ ก่อนจะให้เริ่มค้นหา** 🧹

## หมายเหตุทางเทคนิค

- กุญแจแคชคือ `(id, source_hash, embedder.signature)` ทั้งสามอย่าง — เปลี่ยนอันใดอันหนึ่งถือว่า cache miss
- `_sync` **สร้าง index ใหม่ทุกครั้ง** แม้ไม่มีอะไรเปลี่ยน (`changed == False`) ซึ่งเป็นส่วนเดียวที่ไม่ได้ทำแบบ incremental
- อ้างอิงสเปก: `FR-011`, `SC-004` (ความสอดคล้อง + ประสิทธิภาพ)
